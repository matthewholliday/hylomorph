# MAPS Lite v2 — Minimal Predictive Orchestration

MAPS Lite is a small protocol for parent/child agent delegation that keeps the
predictive loop explicit and machine-checkable:

- **Prior**: parent handoff (goal + assumptions + bounds)
- **Surprise**: child finds evidence that contradicts prior
- **Posterior**: child returns either `result` or `escalation`

Design target: faithful behavior with the fewest moving parts, least parsing
ambiguity, and clear failure containment.

---

## 1) Core design goals

1. Keep each task auditable in one append-only file.
2. Make "wrong world model" failures explicit and fast to escalate.
3. Avoid optional protocol surface unless it prevents real failure modes.
4. Keep the protocol executable with simple YAML parsing and a tiny validator.

Non-goal: rich workflow choreography. This protocol is intentionally narrow.

---

## 2) Single artifact per task

Each delegated task is exactly one Markdown file:

```text
tasks/2026-04-25/retrieve-q3-metrics.md
```

Rules:

- Top of file: exactly one YAML frontmatter block (`handoff`).
- Body: zero or more appended reply sections.
- Replies are immutable; only append new sections.
- Task ends when the latest reply is `result` or `aborted`.

No generated message IDs, no cross-file references, no duplicated metadata.

---

## 3) Minimal schema

### 3.1 Handoff (required at top of file)

```yaml
---
goal: <single sentence outcome>
done_when: <objective completion condition or schema>
assumptions:
  A: <assumption text>
  B: <assumption text>
escalate_if:
  - <condition that must escalate>
limits:
  timeout_s: <int>       # optional; default 60
  max_retries: <int>     # optional; default 1
context: <optional details child cannot derive>
---
```

Required keys: `goal`, `done_when`, `assumptions`, `escalate_if`.

### 3.2 Reply section (appended)

`result` form:

```markdown
## Reply <N> — result
confirmed: [A, B]
artifact: |
  <deliverable matching done_when>
notes: |
  <optional>
```

`escalation` form:

```markdown
## Reply <N> — escalation
violated: [A]
observed: |
  <raw evidence, not paraphrase>
proposal: |
  <optional suggested handoff delta>
disposition: blocked
```

`handoff delta` form (parent only, after escalation):

```markdown
## Reply <N> — handoff
assumptions:
  A: <updated assumption text>
limits:
  max_retries: 2
```

---

## 4) Invariants that prevent drift

These are the hard rules that keep behavior faithful:

1. **Assumption closure**: on `result`, every current assumption ID must appear
   in `confirmed`.
2. **Escalation grounding**: on `escalation`, `violated` must be non-empty and
   `observed` must contain raw evidence.
3. **Delta merge only**: `handoff` replies may only include keys from the
   original handoff schema; merge rule is last-write-wins per key.
4. **Retry isolation**: retries happen only inside child execution and must
   respect `limits.max_retries`; retries do not create reply sections.
5. **Single terminal state**: once `result` or `aborted` exists as latest
   reply, no further sections may be appended.

---

## 5) Parent and child behavior (small state machine)

```text
parent writes handoff
loop:
  child executes within limits
  if done_when satisfied and no assumptions contradicted:
    append result; stop
  else:
    append escalation
    parent either:
      a) appends handoff delta and continue
      b) appends aborted and stop
```

Parent responsibilities:

- Keep `done_when` objective and testable.
- Keep assumptions minimal (only failure-critical beliefs).
- Revise by delta, not rewrite.

Child responsibilities:

- Retry first if success is still plausible under current assumptions.
- Escalate immediately when assumptions are contradicted.
- Include concrete evidence in `observed`.

---

## 6) Defaults chosen for safety and simplicity

If omitted, runtime defaults are:

```yaml
limits:
  timeout_s: 60
  max_retries: 1
```

Why these defaults:

- One retry catches transient errors without hiding persistent model mismatch.
- 60s timeout prevents runaway work for wrong priors.

---

## 7) Worked example (short)

```markdown
---
goal: Retrieve Q3 metrics for Project X.
done_when: JSON with keys `revenue`, `churn_rate`, `burn_rate` in USD.
assumptions:
  A: Data is in `legacy_finance`.
  B: Credential `FIN_SEC_99` has read access.
escalate_if:
  - Any assumption is observed false.
---

## Reply 1 — escalation
violated: [A, B]
observed: |
  `legacy_finance/q3_metrics` returned 410 Gone (moved to cloud_fin_v2).
  `FIN_SEC_99` returned 401 Unauthorized on cloud_fin_v2.
proposal: |
  Update data source to cloud_fin_v2 and credential to OAuth token.
disposition: blocked

## Reply 2 — handoff
assumptions:
  A: Data is in `cloud_fin_v2`.
  B: Env `CLOUD_FIN_TOKEN` has read access.

## Reply 3 — result
confirmed: [A, B]
artifact: |
  {"revenue":1250000,"churn_rate":0.024,"burn_rate":310000,"currency":"USD"}
```

---

## 8) Tiny validation contract (recommended)

Minimum validator checks:

- Handoff required keys exist.
- Reply header type is one of `result`, `escalation`, `handoff`, `aborted`.
- `result` requires `confirmed` + `artifact`.
- `escalation` requires `violated` + `observed` + `disposition`.
- `violated`/`confirmed` IDs must exist in current assumptions.

If validation fails, treat as protocol error and append:

```markdown
## Reply <N> — aborted
reason: protocol_validation_failed
```

Fail closed, not silent.
